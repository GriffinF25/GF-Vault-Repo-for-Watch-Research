## 2026-07-16 harvest

1. **Claude Code Dynamic Workflows reach GA with adversarial verification** — a workflow is a script (agent-authored) that fans a task out to many subagents, with independent agents adversarially reviewing each other's findings before reporting; unverifiable claims get labeled "unverified" instead of silently dropped. Source: Anthropic, Claude Code Docs, "Orchestrate subagents at scale with dynamic workflows" (code.claude.com/docs/en/workflows), GA July 2, 2026; corroborated by InfoQ, June 2026. **Upgrade:** codify the workspace's produce→measure→critic loop as a saved `.claude/workflows/` script so it runs as a resumable, auditable pipeline instead of turn-by-turn orchestration — this is the `Workflow` tool already available in this session.

2. **Anthropic's `/goal` productizes writer/verifier separation** — sets a completion condition; after every turn a separate small/fast model (can't call tools, sees only the transcript) judges pass/fail against the condition and either stops Claude or feeds back a reason to continue. Source: Anthropic, Claude Code Docs, code.claude.com/docs/en/goal (requires v2.1.139+, shipped within ~last 1-2 months). **Upgrade:** mirror this as a project Stop hook so the critic-agent gate becomes a deterministic check on every orchestrator turn for high-stakes tasks, not a step that has to be remembered.

3. **Model-tiering via a standing "use your judgement" instruction, not a fixed table** — Jesse Vincent's tip, popularized by Simon Willison: telling the top-tier model "for all coding tasks, use your judgement to pick an appropriate lower-power model and run it in a subagent" reliably delegates mechanical work down without a hand-authored routing table, and visibly cuts token spend. Source: Simon Willison, simonwillison.net, "Fable's judgement," July 3, 2026 (x.com/simonw/status/2073117641020215566); originated with Jesse Vincent. **Upgrade:** add one line to `agent-patterns.md`'s delegation rules — default to judgement-based tier selection per subtask, falling back to the fixed analyst=sonnet/scout=haiku table only when ambiguous.

4. **Diff-only adversarial reviewers, blind to the implementer's reasoning** — in Bun's 535K-line Zig→Rust rewrite (11 days, ~64 concurrent agents in worktree shards), every fix went through 1 implementer → 2 independent reviewers who saw *only the diff* (not the implementer's rationale) and were told to assume it's broken → 1 fixer. Caught real bugs (e.g. use-after-free) pre-merge. Source: Jarred Sumner, "Rewriting Bun in Rust," bun.com/blog/bun-in-rust, July 8, 2026; corroborated by Simon Willison (simonwillison.net/2026/Jul/8/rewriting-bun-in-rust/) and HN (news.ycombinator.com/item?id=48837877). **Upgrade:** harden the builder→critic loop — for high-stakes changes, strip the critic's prompt of the builder's justification and give it only the diff/output, matching the workspace's existing "producer self-verification has self-grading bias" doctrine but tightened.

5. **Anthropic's decision framework for the 7 instruction layers (CLAUDE.md vs. skills vs. hooks vs. subagents)** — core rule: CLAUDE.md is for always-on facts, skills are for reusable procedures, and any "every time X, always do Y" line in CLAUDE.md should become a hook instead, since hooks fire deterministically while prose can be deprioritized under context pressure. Source: Anthropic/Claude Code team, "Steering Claude Code," claude.com/blog/steering-claude-code-skills-hooks-rules-subagents-and-more, ~late June/July 2026. **Upgrade:** audit root and domain CLAUDE.md files for deterministic "always do Y" lines (e.g. no-bloat file-creation checks) and convert the enforceable ones into `.claude/settings.json` hooks instead of relying on prose the orchestrator might drop under context pressure.

6. **Verifiers/comparators need to co-evolve with the generator, not stay fixed** — as producer quality rises, static graders become the bottleneck and get gamed (reward hacking against the verifier itself); verification must be periodically re-evaluated on scalability, faithfulness, and robustness, not treated as a permanent fixture. Source: Qwen Team (Alibaba), "The Verification Horizon: No Silver Bullet for Coding Agent Rewards," arXiv:2606.26300, submitted June 24, 2026. **Upgrade:** add a line to `agent-patterns.md`'s verification-loop pattern — the critic gate should occasionally spot-check whether the comparator itself (screenshot-diff, test suite, pricing comps) is still a faithful measure as producer agents get better at satisfying it, not just apply it blindly.

7. **Merge redundant sequential reviewer subagents into one multi-verdict pass** — Jesse Vincent found that combining two separate reviewer subagents (spec-compliance + code-quality) into one that returns both verdicts in a single pass, plus pre-baking the review "packet" so reviewers rarely shell out to git themselves, cut token spend ~60% and wall-clock ~50% on real benchmark tasks. Source: Jesse Vincent, "Superpowers 6 — Massively Parallel Procrastination," blog.fsck.com/2026/06/15/Superpowers-6/, June 15, 2026; discussed on HN (news.ycombinator.com/item?id=48739459). **Upgrade:** audit any workspace pattern that dispatches two sequential subagents for related checks and consolidate into one subagent returning a combined Result/Evidence/Confidence report — cheaper and faster without losing the check.

8. **Regression-safe LLM-as-judge evals: trajectory checkpoints + blocking gate on regression** — treat the judge as a calibrated component, not an oracle; write checkpoints for which tool calls/params must occur (not just output assertions); make "correct" explicit via an inclusion/exclusion rubric; run the eval suite on every prompt/agent-spec change and block if scores regress. Source: TechEmpower Blog, "Evals for Agentic Loop Applications," techempower.com/blog/2026/07/14/evals-for-agentic-loop-applications/, July 14, 2026. **Upgrade:** extend the upgrade-candidates.md pipeline's "test on a real task" step with a small fixed task set + explicit rubric per agent definition, so promoting a change to `.claude/agents/*.md` requires a pass/fail comparison against the prior version, not just a spot-check.

1. **Subagents no longer nest by default; new spawn/concurrency caps** — between v2.1.211–v2.1.217 (July 15–21, 2026), Claude Code added three governors on subagent fan-out: a per-session spawn cap (`CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSION`, default 200), a concurrency cap (`CLAUDE_CODE_MAX_CONCURRENT_SUBAGENTS`, default 20), and — most relevant here — subagents no longer spawn nested subagents by default (requires raising `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH`). `--max-budget-usd` now also halts running background subagents, not just new spawns. Source: Claude Code official changelog, code.claude.com/docs/en/changelog, v2.1.211/212/217, July 15–21, 2026 (direct fetch). **Upgrade:** `agent-patterns.md`'s "keep the tree shallow... no deeper" is now platform-enforced by default, not just doctrine — add a line noting tier-1 specialists (analyst/builder/site-cloner/critic) cannot delegate further to `scout` without explicitly raising `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH`, so an attempted fan-out silently fails to nest rather than working as assumed.

2. **`/fork` now forks the whole session; `/subtask` is the new name for in-session delegation; subagents inherit the parent's permission mode** — v2.1.212 (July 17) split `/fork` (now copies the entire conversation into a parallel background session) from `/subtask` (the old `/fork` behavior — launch an in-session subagent). The Task tool's `mode` parameter was deprecated: subagents now inherit the parent session's permission mode by default instead of an explicit override. Source: Claude Code official changelog, v2.1.212, July 17, 2026 (direct fetch). **Upgrade:** `critic.md` is read-only by design via tool-list restriction alone (no Edit/Write granted) — under this change it also inherits the session's permission mode (e.g. `acceptEdits` per `.claude/settings.json`), and its Bash grant is broad under that mode. Add an explicit permission-mode note to `critic.md` so its read-only guarantee doesn't rest on the tool list alone. Also add one line to Pattern 1 distinguishing `/subtask` (the tree's actual delegation primitive) from `/fork` (parallel-session explore, out of scope for current tree doctrine).

3. **Skills with `context: fork` now run in the background by default** — v2.1.218 (July 22) made any skill whose frontmatter sets `context: fork` (i.e., it runs as an isolated subagent, with `agent:` picking which subagent type executes it) run in the background automatically unless it opts out with `background: false`. Source: Claude Code official changelog, v2.1.218, July 22, 2026, and Skills docs "Run skills in a subagent" section (direct fetch). **Upgrade:** the workspace's No-Bloat rule already says a recurring workflow becomes a skill — this is the concrete mechanism to do that without spending main-session context: a repeatable analyst-style task (e.g., a GF Vault pricing-comp pull) can become `gf-vault/.claude/skills/<name>/SKILL.md` with `context: fork` + `agent: analyst`, running background-by-default and directly reinforcing the "orchestrator context is the scarcest resource" rule.

4. **`/verify` and `/code-review` no longer self-invoke** — v2.1.215 (July 19) changed these two bundled skills so Claude can no longer trigger them autonomously mid-conversation; they only run when explicitly invoked, removing a safety net that used to fire opportunistically. Source: Claude Code official changelog, v2.1.215, July 19, 2026 (direct fetch). **Upgrade:** if the orchestrator has been counting on built-in `/verify`/`/code-review` firing as a supplement to the `critic` agent gate in Pattern 2, that no longer happens automatically — add a line noting these must now be explicitly invoked alongside (not instead of) `critic`.

5. **Sonnet 5's new tokenizer inflates token counts ~30% vs. Sonnet 4.6, for identical text** — Sonnet 5 (this session's own model) inherited Opus 4.7's tokenizer, which is measurably less efficient per character. Per-token pricing is unchanged, so this is a silent capacity/cost regression: any token-budget assumption calibrated on 4.6 (`max_tokens` limits, compaction triggers, iteration caps) is now miscalibrated with no code change. Source: Anthropic, Claude Platform Docs "Migration guide," platform.claude.com/docs/en/about-claude/models/migration-guide (direct fetch, exact quote verified); corroborated by dev.to/synthorai and MarkTechPost, July 13, 2026. **Upgrade:** `agent-patterns.md`'s "~60 lines" split threshold and the 5-pass iteration cap were both likely sized against pre-Sonnet-5 token math — re-baseline rather than assume old thresholds still hold, since this workspace's own sessions now run on Sonnet 5.

6. **Anthropic's "loop engineering" taxonomy: four loop primitives by what you hand off** — Claude Code's team formalized Boris Cherny's "loop engineering" concept into four primitives distinguished by what gets handed to the loop: turn-based (hands off the check — a skill judges done/not-done each turn), goal-based (hands off the stop condition — `/goal`, already known to this workspace), time-based (hands off the trigger — `/loop`/`/schedule`, runs on a schedule regardless of goal state), and proactive (hands off the whole prompt). Also introduces `/usage` for breaking down token spend by skill/subagent/MCP. Source: Anthropic, claude.com/blog/getting-started-with-loops, July 7, 2026 — direct fetch blocked by proxy this session; corroborated across 4+ independent secondary write-ups (explainx.ai, mcp.so, mer.vin, The New Stack) with consistent quotes and detail. **Upgrade:** this very AI Practice Scout routine is a time-based loop by this taxonomy, not goal-based, and is currently just a remembered schedule — consider attaching `/usage`-style cost accounting to Pattern 2's iteration cap, so a verification loop that plateaus but keeps burning tokens gets flagged rather than just capped at 5 passes.

7. **"When Agents Do Not Stop": nested agent-handoff loops are a distinct failure mode from top-level loops** — a new paper categorizes agent non-termination into four failure modes (repeated model calls, repeated tool calls, workflow-transition loops, and agent-handoff loops — nested subagent delegation re-triggering itself) and argues stopping conditions must be layered (iteration cap + cost budget) and propagated across nested handoffs, not just enforced at the top level. Source: arXiv:2607.01641, "When Agents Do Not Stop: Uncovering Infinite Agentic Loops in LLM Agents," Hou/Wang/Zhao/Wang, submitted July 2, 2026 — corroborated via the arXiv abstract page across independent search queries; direct fetch blocked by proxy this session. **Upgrade:** Pattern 2's 5-pass/plateau cap currently reads as a single top-level check. Tighten it: when a tier-1 specialist itself delegates further (subject to item 1's new nesting-depth limits), the cap should apply per handoff chain, not just at the orchestrator's top level.

8. **Mollick/Wharton: prompting tricks show near-zero effect on frontier models — explicit specs and evaluation criteria are what move output quality** — Wharton's Generative AI Labs ran controlled tests of chain-of-thought prompting, threats/"tipping," and expert personas against frontier models on hard benchmarks; all showed near-zero or unreliable effect (individual questions swing ±35% with no net direction) on models that already reason internally by default. Synthesis: treat the model as a literal executor needing explicit goals, output definitions, and evaluation criteria, not prompt aesthetics. Source: Ethan Mollick (Wharton, Generative AI Labs co-director), X post July 7, 2026, referencing Wharton GAI Lab technical reports (e.g. "Playing Pretend: Expert Personas Don't Improve Factual Accuracy," SSRN 5879722) — corroborated via explainx.ai and Blockchain.News summaries; direct fetch blocked by proxy this session. **Upgrade:** validates rather than changes existing doctrine (the delegation rules already require objective/inputs/done-criteria/report-format) — concrete action is to audit domain specs like `gf-vault/agents/watch-pricing-genius.md` for persona/CoT-inducing phrasing this research suggests is dead weight, and replace it with an explicit rubric, which is exactly what item 8 of the July 16 harvest (TechEmpower trajectory checkpoints) would provide.
